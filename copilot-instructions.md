High-level architecture (big-picture)
- The repository hosts a web frontend (web/) implementing a platform that ingests smart-contract source + ABIs and emits a dashboard config JSON.
- Routes of interest (conventionally expected by the app):
  - / (Landing)
  - /submit (multi-step upload UI for ABIs, source, addresses, brand)
  - /jobs/[id] (Build status page: multi-phase build console)
  - /dashboard/[id] (Generated dashboard rendered from server-emitted JSON)
- Responsibilities:
  - Server: accept uploads, persist jobs, run the analysis pipeline (parsing, archetype classification, parameter extraction, fetch-plan generation, brand extraction, layout/token emission) and return a single config JSON keyed by job id.
  - Client: drive the upload UI, instantiate ethers.js v6 Contracts, batch on-chain reads via multicall3, and render the dashboard consuming `design_tokens` + layout config.
- Important libraries: next@16.x, react@19, ethers@6.x, multicall (aggregate3 patterns implied), tailwindcss, shadcn UI.

Key conventions and patterns (codebase-specific)
- Ethers v6 is required: all example read code and multicall payloads must be v6-compatible (do not generate v5-style code).
- Multicall batching: group reads into multicall3 aggregate3-style batches. Prefer multicall over naive per-call RPC patterns.
- Design tokens / layout JSON: the backend emits a structured `design_tokens` + `layout` payload. The frontend consumes these by name (colors, typography, radii, shadows, chart palette). Do NOT hardcode colors, fonts, radii, RPC URLs, or contract addresses in layout components — reference token names instead.
- Upgradeable-proxy handling: the fetch plan must detect common proxy patterns (EIP-1967, Beacon, Diamond) and resolve implementation ABIs at runtime.
- Security / scope: repository is EVM-only (Ethereum + EVM L2s). Non-EVM chains are out-of-scope; code and generated fetch plans should reflect that.
- Build-job phases: the build console models discrete phases (Parsing, Classification, Parameter extraction, Fetch-plan generation, Brand extraction, Layout emission). Treat them as distinct, observable states in `/jobs/[id]`.
- WCAG & color rules: chart palettes and text-on-bg pairs must pass contrast checks; prefer colorblind-safe palettes.
